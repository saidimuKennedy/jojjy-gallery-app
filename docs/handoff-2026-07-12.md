## Critical ground rule: the database

`DATABASE_URL`/`DIRECT_URL` in `.env` point at a **live managed Postgres**
(`db.prisma.io`) with real seeded data. **Never run `prisma migrate dev`
against it** ‚Äî it can prompt/auto-reset and wipe data. The safe command to
apply pending migrations is `prisma migrate deploy` (applies only, never
resets) ‚Äî still has not been run against the real DB in this session.

The established pattern for adding *any* further schema change:
1. Edit `prisma/schema.prisma`, run `npx prisma format` then
   `npx prisma validate`.
2. Hand-write the migration SQL yourself in a new
   `prisma/migrations/<timestamp>_<name>/migration.sql` ‚Äî do **not** run
   `migrate dev` to generate it.
3. Verify it by spinning up a throwaway **local** Postgres cluster (this
   box has `initdb`/`pg_ctl`/`createdb` available, no Docker needed ‚Äî see
   any earlier migration folder's construction in this session for the
   exact recipe: `initdb` into a scratchpad dir, start on a free port with
   a short-path unix socket dir, `createdb`, then point `DIRECT_URL` at it
   via env override ‚Äî `prisma.config.js` reads `DIRECT_URL`, not
   `DATABASE_URL`, for CLI/migrate operations).
4. `prisma migrate deploy` against that throwaway DB (replays full
   history), then `prisma migrate diff --from-config-datasource
   prisma.config.js --to-schema prisma/schema.prisma --exit-code` ‚Äî must
   print "No difference detected" / exit 0. If it shows a diff, fix the
   hand-written SQL (common gotcha hit twice already: Prisma's default
   `onDelete` for an **optional** relation is `SET NULL`, not `RESTRICT` ‚Äî
   get this wrong and the diff check will catch it).
5. Tear down the throwaway cluster (`pg_ctl ... stop -m fast`, `rm -rf`
   the data/socket dirs).
6. `npx prisma generate` to refresh the client.

Only after that rigor should `prisma migrate deploy` ever be run against
the real `DIRECT_URL` ‚Äî and that's still a decision for the user, not to
be done unprompted.
